# Specialist Response Contract

## Purpose

This contract defines what a good manual specialist response should look like after a user copies a generated specialist prompt into an assistant.

The goal is not automatic import yet. The goal is to make specialist responses:

- easy for a human to read and approve
- consistent across UX/UI, Front-end, Back-end, QA, and Tech Lead roles
- structured enough for future examples and a future local checker
- aligned with the local-first V1 workflow

This contract applies to responses produced from specialist prompts generated by:

- `pnpm prompt:specialist`
- `pnpm prompt:specialists`

## Response Format

The response should be Markdown.

It should be concise, review-ready, and organized by backlog item ID when relevant.

It should not:

- assume direct file editing
- assume automatic backlog import
- assume remote issue creation
- assume autonomous approval or implementation

## Expected Markdown Response Structure

A strong response usually follows this shape:

```md
# Specialist Response

## Role

<specialist role>

## Scope

- backlog items reviewed
- short summary of what was analyzed

## Item Notes

### <backlog-item-id> <optional short title>

- findings, notes, or checks
- role-specific recommendations

## Proposed Tasks Or Recommendations

- task or recommendation linked to a backlog item ID

## Assumptions

- explicit assumptions

## Open Questions

- explicit unanswered questions

## Risks

- explicit risks

## Suggested Backlog Updates

- suggested human-reviewed updates without directly editing files
```

The exact wording can vary, but the sections and intent should stay recognizable.

## Required Sections

These sections should always be present:

### `# Specialist Response`

Use a top-level heading that clearly identifies the document as a specialist response.

### `## Role`

State the acting role explicitly using one of:

- `ux-ui`
- `frontend`
- `backend`
- `qa`
- `tech-lead`

### `## Scope`

State which backlog items were reviewed and what kind of review was performed.

At minimum, this section should include:

- one or more backlog item IDs
- a short summary of the review focus

### `## Item Notes`

This is the main section.

Use one subsection per relevant backlog item, for example:

```md
### story-002 Homepage value proposition
```

or:

```md
### task-004
```

Each item note should stay linked to the backlog item being discussed.

### `## Assumptions`

List the assumptions the specialist had to make.

If there are no assumptions, say:

```md
- No new assumptions.
```

### `## Open Questions`

List unanswered questions that block safe review or implementation planning.

If there are no open questions, say:

```md
- No open questions.
```

### `## Risks`

List delivery, quality, sequencing, accessibility, regression, or technical risks relevant to the role.

If there are no meaningful risks, say:

```md
- No major new risks identified.
```

### `## Suggested Backlog Updates`

List recommended backlog changes for human review.

This section should describe proposed edits, additions, or clarifications without directly editing files.

If there are no suggested updates, say:

```md
- No backlog updates suggested.
```

## Optional Sections

These sections are useful when needed, but not required every time:

- `## Summary`
- `## Recommended Next Step`
- `## Acceptance Coverage Notes`
- `## Dependency Notes`
- `## Test Checklist`
- `## Sequencing Notes`
- `## Accessibility Notes`

Optional sections should still stay tied to the role and reviewed backlog items.

## How To Reference Backlog Item IDs

Always reference backlog item IDs explicitly.

Good examples:

- `story-002 needs clearer acceptance coverage for error states.`
- `task-004 should be split into two front-end tasks.`
- `risk-005 already covers scope drift, but it should mention content approval timing.`

Avoid vague references such as:

- `this story`
- `the task above`
- `the item`

If a recommendation creates a new task suggestion, mention the parent item ID when relevant.

Example:

```md
- Suggested task under story-002: add a front-end task for empty-state messaging.
```

## How To Reference The Specialist Role

The role should be stated exactly in the `## Role` section.

Recommended format:

```md
## Role

frontend
```

The rest of the response should stay consistent with that role.

For example:

- `frontend` should focus on UI implementation planning, component concerns, and integration notes
- `qa` should focus on test coverage, blockers, and regression risk
- `ux-ui` should focus on flow clarity, content, usability, and accessibility concerns
- `backend` should focus on API, data, service, and dependency concerns
- `tech-lead` should focus on sequencing, cross-role risk, and architecture-aligned delivery review

## How To Express Proposed Tasks Or Recommendations

Keep proposed tasks or recommendations explicit and reviewable.

Recommended format:

```md
- story-002: recommend adding a front-end task for loading, empty, and error states.
- task-004: recommend splitting implementation planning from content dependency follow-up.
```

Use verbs such as:

- `recommend adding`
- `recommend clarifying`
- `recommend splitting`
- `recommend updating`
- `recommend sequencing after`

Do not pretend the task already exists unless it does.

## How To Express Assumptions

Assumptions should be clearly labeled as assumptions, not facts.

Good examples:

- `Assumption: story-002 will be implemented in a single responsive page flow.`
- `Assumption: task-004 will use the current front-end stack already approved by the team.`

Recommended list format:

```md
- Assumption: <statement>
```

## How To Express Open Questions

Open questions should point to missing information, unclear decisions, or blocking uncertainty.

Good examples:

- `Question: Does story-002 require localization in V1?`
- `Question: Should QA validate only desktop browsers in MVP scope?`

Recommended list format:

```md
- Question: <statement>
```

## How To Express Risks

Risks should describe the delivery concern and why it matters.

Good examples:

- `Risk: story-002 does not define fallback content, which could block safe front-end implementation.`
- `Risk: task-004 depends on content decisions that are not yet reviewed, which may cause rework.`
- `Risk: missing acceptance criteria makes QA coverage subjective and hard to approve.`

Recommended list format:

```md
- Risk: <statement>
```

## How To Express Suggested Backlog Updates

Suggested backlog updates should remain descriptive and human-controlled.

Good examples:

- `Recommend updating story-002 acceptance criteria to include loading, empty, and error states.`
- `Recommend adding a QA task under story-002 for regression checks on the main lead form flow.`
- `Recommend adding a risk item for dependency on final homepage copy approval.`

Do not write as if the update has already been applied.

Avoid:

- `Updated story-002 acceptance criteria`
- `Created task-009`
- `Marked task-004 ready`

Prefer:

- `Recommend updating`
- `Recommend adding`
- `Recommend clarifying`
- `Recommend splitting`

## What Specialists Must Not Do

Specialists must not:

- implement code
- claim to have edited backlog files directly
- claim to have changed statuses automatically
- mark work approved or ready automatically
- invent external APIs, automation, or remote issue workflows
- silently change scope owned by another role
- replace explicit backlog IDs with vague references
- present assumptions as confirmed facts

## Role Alignment Notes

The response should stay aligned with the current role templates:

- `ux-ui` focuses on flows, structure, accessibility, and content gaps
- `frontend` focuses on UI implementation planning, components, state, and integration concerns
- `backend` focuses on APIs, services, data, and technical constraints
- `qa` focuses on test coverage, blockers, regression risks, and acceptance checks
- `tech-lead` focuses on sequencing, dependencies, solution risk, and architecture-aligned review

If a role crosses into another role's area, it should call that out as a recommendation or question instead of acting as that other role.

## Example: Front-end Response

```md
# Specialist Response

## Role

frontend

## Scope

- Reviewed story-002 and task-004.
- Focused on UI implementation planning, state handling, and content dependencies.

## Item Notes

### story-002 Homepage value proposition

- The story is implementable, but the acceptance criteria do not mention loading, empty, or error states.
- The current description suggests one main CTA flow, but the expected fallback behavior is not defined.

### task-004 Prepare homepage implementation tasks

- The task is directionally useful, but it mixes page structure work with implementation planning.
- It would be easier to estimate if content dependency follow-up were separated from UI delivery planning.

## Proposed Tasks Or Recommendations

- story-002: recommend adding a front-end task for loading, empty, and error states.
- story-002: recommend clarifying CTA behavior for mobile and desktop breakpoints.
- task-004: recommend splitting implementation planning from content dependency tracking.

## Assumptions

- Assumption: story-002 will be delivered as a single responsive homepage flow.
- Assumption: content structure from UX/UI review will be available before UI implementation starts.

## Open Questions

- Question: Is localization out of scope for the homepage in V1?
- Question: Should the CTA submit flow support inline validation only, or summary errors too?

## Risks

- Risk: missing state handling criteria may cause inconsistent implementation decisions.
- Risk: unresolved content dependencies may create avoidable front-end rework.

## Suggested Backlog Updates

- Recommend updating story-002 acceptance criteria to include loading, empty, error, and validation behavior.
- Recommend adding a front-end task under story-002 for responsive CTA state handling.
```

## Example: QA Response

```md
# Specialist Response

## Role

qa

## Scope

- Reviewed story-002 and task-004.
- Focused on acceptance coverage, testability, and regression risk.

## Item Notes

### story-002 Homepage value proposition

- The story intent is clear, but acceptance coverage is incomplete for negative and edge-case behavior.
- There is no explicit acceptance point for CTA validation, broken content states, or browser coverage.

### task-004 Prepare homepage implementation tasks

- The task needs clearer links to the acceptance criteria it supports.
- QA planning is blocked until final user-visible states are defined.

## Proposed Tasks Or Recommendations

- story-002: recommend adding a QA task for regression checks on the homepage CTA flow.
- story-002: recommend clarifying expected behavior for invalid input, missing content, and submission errors.

## Assumptions

- Assumption: QA review in V1 will be manual and checklist-based.
- Assumption: the homepage CTA is part of MVP scope, not a future enhancement.

## Open Questions

- Question: Which browsers or devices are considered in-scope for V1 review?
- Question: Should QA validate accessibility basics for the homepage CTA flow in V1?

## Risks

- Risk: incomplete acceptance criteria makes pass/fail review subjective.
- Risk: undefined fallback behavior increases regression risk when implementation begins.

## Suggested Backlog Updates

- Recommend updating story-002 acceptance criteria to include invalid input and failed submission behavior.
- Recommend adding a QA task under story-002 for manual regression coverage of the homepage lead flow.
```
